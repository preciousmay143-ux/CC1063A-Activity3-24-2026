Laboratory 1: Tools & Architecture Setup
Project Purpose this repository is built for Laboratory 1: Tools & Architecture Setup and the goal is to First set up essential development tools likeVS Code as the code editor and Git for version control. Second create a well-organized structure that keeps code easy to find, update, and scale and the Last is to define clear boundaries for different parts of the project to avoid confusion as work progresses. And our chosen folder structure are 1.src/ - it holds all working code for the project this is where we build our solution. 2.src/components - its stores reusable user interface/UI parts for example like buttons, forms, cards and these only handle how things look and how users interact with them no behind the scenes work here. 3.src/logic/ - it manages core project rules and workflows for example calculating values and validating inputs. This layer connects what the user sees to how data is handled. 4.src/data/ - deals with all data-related tasks like reading from files, connecting to APIs, storing information. It hides complex data work from other parts of the project. 5.docs/ - it contains extra guides, diagrams, or notes that help explain the project like how to set up the environment and design decisions. 6.tests/ - has files to check if our code works correctly like testing if a button triggers the right action and if calculations are accurate. We chose this structure because it groups similar work together so anyone working on the project can immediately find where to add or change code and it also prevents one part of the project from interfering with another. The component layering strategy we use a 3 layer system to separate different types of work each layer has one distinct job and communicates clearly with the others so the Presentation Layer (src/components/) is responsible for showing information to the user and responding to their actions like clicks or typing it follows a strict rule of not handling any business logic or data processes its only displaying content it receives and forwarding user actions to the next layer for example a Submit Form button that sends input to the logic layer without verifying its correctness. Next is the Business Logic Layer (src/logic/) it focuses on making decisions and processing information according to project rules it operates independently of how data is stored or how the UI is designed, concentrating solely on defining what the project needs to do such as checking if user input meets set requirements before it is sent for saving. And the last one is the Data Layer (src/data/) it manages where data is stored and how it is retrieved or updated its only role is handling data-related tasks and returning results to the logic layer, without making judgments about what the data represents an example being saving valid user input to a file or fetching information from an online source. We chose this strategy because it isolates responsibilities updating the UI’s appearance only requires changes to the presentation layer, while switching data storage systems only affects the data layer, which streamlines modifications and minimizes errors.
